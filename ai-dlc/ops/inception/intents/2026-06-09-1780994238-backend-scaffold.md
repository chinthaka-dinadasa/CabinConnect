# Intent: Backend Scaffold — .NET 8 API with Supabase JWT, PostgreSQL, and Health Check

**Status:** Elaborated
**Date:** 2026-06-09
**Owner:** Chinthaka

---

## What

Create the CabinConnect .NET 8 Web API backend project from scratch — configured with Supabase JWT validation middleware so that only requests carrying a valid Supabase-issued access token can reach protected endpoints, a Supabase PostgreSQL database connection via Npgsql/EF Core so the data layer is ready for feature bolts, and a publicly accessible health check endpoint that verifies both API availability and database connectivity. This is the foundational backend capability — all future feature bolts (Cabin management, Booking API, etc.) depend on it.

## Why

The frontend is operational and can generate valid Supabase JWTs, but there is no .NET API yet to call. Before any business feature can be built, the API project must exist, JWT authentication must be wired up correctly, and there must be a baseline endpoint to confirm the service is running. This intent delivers exactly that foundation — nothing more.

## Success Looks Like

- The .NET 8 Web API starts without errors and listens on a configurable port
- A request to `GET /health` returns `200 OK` without an `Authorization` header, and the response confirms both the API and the database connection are healthy
- A request to any protected endpoint without a JWT returns `401 Unauthorized`
- A request to any protected endpoint with a valid Supabase-issued JWT returns a non-401 response
- A request with an expired or tampered JWT returns `401 Unauthorized`
- The application can open a connection to the Supabase PostgreSQL database on startup
- All configuration (Supabase URL, JWT secret, database connection string, allowed CORS origins) is read from environment variables — no secrets in source code

## Assumptions

- Supabase project is already created and the JWT secret is available in the Supabase dashboard (Settings → API → JWT Secret)
- The Supabase PostgreSQL connection string (host, port, database, user, password) is available in the Supabase dashboard (Settings → Database → Connection string)
- The .NET 8 SDK is installed in the development environment
- EF Core with Npgsql provider is used for database access; no migrations are run in this intent — the connection is established and verified only
- The frontend runs on `http://localhost:5173` (Vite default) — this is the CORS origin to allow in development
- Supabase issues HS256 JWTs; the JWT secret is used for symmetric validation
- The API will run on `http://localhost:5000` in local development (matching `VITE_API_BASE_URL` placeholder set in the frontend bolt)
- RLS is enforced on all Supabase tables; the backend connects as the service role for server-side queries

## Open Questions

- None. Scope is clearly bounded to scaffold only.

## Out of Scope

- Any business endpoints (Cabins, Bookings, Guests, Hosts)
- Database migrations or schema setup — tables are created in a future intent
- Entity models and repository pattern implementation — deferred to feature bolts
- Supabase JS client integration — the backend connects to PostgreSQL directly via Npgsql, not via the Supabase SDK
- Deployment configuration (Docker, cloud hosting)
- Rate limiting or logging middleware beyond the default .NET request logging
- Role-based authorisation (beyond authenticated vs. unauthenticated)
