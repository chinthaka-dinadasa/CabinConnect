# Improvement: .env values with semicolons or spaces must be double-quoted

**Status:** Applied
**Triggered by:** [ops/operate/retros/2026-06-09-1781002300-cabin-registration.md](../retros/2026-06-09-1781002300-cabin-registration.md)
**Applied date:** 2026-06-09

---

## Target File

`backend/.env.example`

---

## Current Text

```
# PostgreSQL connection string — Settings → Database → Connection string
# Use keyword format with SSL Mode=Require (added in Unit 3)
# DATABASE_URL=Host=...;Port=5432;Database=postgres;Username=postgres;Password=...;SSL Mode=Require
```

---

## Proposed Replacement

```
# PostgreSQL connection string — Settings → Database → Connection string
# Use keyword format. Values containing ; or spaces MUST be double-quoted.
# DATABASE_URL="Host=...;Port=5432;Database=postgres;Username=postgres;Password=...;SSL Mode=Require"
```

---

## Reason

During the Cabin Registration bolt, `DATABASE_URL` was set without double-quotes. When sourced via `set -a && source .env && set +a`, bash treated the semicolons in the connection string as command separators, silently truncating the value to just `Host=db.xxx.supabase.co`. `echo $DATABASE_URL` showed the truncated value but the error from `dotnet ef` pointed to the wrong cause, leading to two diagnostic steps before the root cause was identified. Showing double-quotes in the example prevents this class of silent failure for every engineer who copies the file.

---

## Validation

- [ ] The next engineer who sets up the backend uses the quoted format on the first attempt
- [ ] `echo $DATABASE_URL` after `source .env` returns the full connection string including port, password, and SSL mode
